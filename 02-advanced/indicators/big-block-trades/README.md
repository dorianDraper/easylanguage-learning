# Big Block Trade Activity — v1.0 & v2.0

🇺🇸 English | 🇪🇸 [Español](README.es.md)

## System Description

Big Block Trade Activity is a real-time microstructure indicator that measures the proportion of intrabar tick activity generated by large price updates — a proxy for institutional or large-order activity within the current bar. The indicator operates exclusively on the live bar, using `IntrabarPersist` variables to accumulate state across tick-by-tick updates without recalculating historical bars.

On each tick, the indicator estimates the size of the most recent trade by computing the delta between the current and previous cumulative tick count. If this delta meets or exceeds the defined block trade threshold, its contribution is added to a running accumulator. The ratio of block activity to total tick count is then expressed as a percentage and plotted continuously. An alert fires when this percentage exceeds a configurable threshold, signaling that large-order activity is dominating the bar.

A companion reference line plots the alert threshold directly on the indicator subgraph, providing a permanent visual baseline for interpreting the block activity percentage.

## Core Mechanics

### 1. Intrabar State Management

All calculation variables are declared as `IntrabarPersist`:

```pascal
Vars:
    IntrabarPersist CurrentTradeVolume(0),
    IntrabarPersist PreviousTickCount(0),
    IntrabarPersist BigBlockVolume(0),
    IntrabarPersist BigBlockPercentage(0);
```

`IntrabarPersist` variables retain their value between consecutive tick updates within the same bar. Without this modifier, EasyLanguage would reset variables to their initial value on every tick, making within-bar accumulation impossible. Think of `IntrabarPersist` as a clipboard that survives each tick update but is explicitly cleared when the bar closes.

The entire calculation block is wrapped in `If LastBarOnChart Then` — ensuring no historical recalculation occurs. The indicator is designed to function in real-time only.

### 2. Tick Delta Estimation

```pascal
If PreviousTickCount > 0 Then
Begin
    CurrentTradeVolume = Ticks - PreviousTickCount;
    ...
End;

PreviousTickCount = Ticks;
```

`Ticks` is an EasyLanguage built-in that returns the cumulative count of price updates on the current bar since it opened. On each new tick, `Ticks` increments by 1. The difference between the current and previous `Ticks` value — `CurrentTradeVolume` — represents the number of price updates that occurred since the last indicator evaluation.

The guard `If PreviousTickCount > 0` prevents a spurious delta on the very first tick of a new bar, when `PreviousTickCount` has just been reset to `0` and no reference point yet exists.

> **Important — what `Ticks` actually measures:** `Ticks` counts **price update events**, not contracts or shares traded. Each time the price changes within the bar, `Ticks` increments by 1, regardless of the volume behind that change. A single institutional order for 2,000 contracts may register as one tick update or several, depending on how the exchange reports the execution. Two small retail orders at the same price may count as a single tick if the price does not change between them. `CurrentTradeVolume` is therefore an approximation of trade activity intensity, not a direct count of contracts. See the Limitations section for a full discussion.

### 3. Big Block Classification

```pascal
If CurrentTradeVolume >= BigBlockTradeSize Then
    BigBlockVolume = BigBlockVolume + CurrentTradeVolume;
```

If the tick delta meets or exceeds `BigBlockTradeSize`, the update is classified as a block trade and its delta is added to the `BigBlockVolume` accumulator. This accumulator grows throughout the bar and is reset only when a new bar opens.

### 4. Percentage Calculation

```pascal
If Ticks > 0 Then
    BigBlockPercentage = (BigBlockVolume / Ticks) * 100;
```

`BigBlockPercentage` expresses what proportion of total tick activity has been classified as block trade activity. The guard `If Ticks > 0` prevents a division-by-zero condition, though in practice `Ticks` will always be positive inside the `If LastBarOnChart` block.

### 5. Alert Logic

```pascal
Plot1(BigBlockPercentage, "BlockPct");

If BigBlockPercentage > AlertThresholdPct Then
    Alert("High block trade activity detected");
```

The percentage is plotted continuously and compared against `AlertThresholdPct`. When block activity exceeds the threshold, an alert fires. The alert is evaluated on every tick, so it may fire multiple times within the same bar if the condition remains true across successive updates.

### 6. Bar Reset

```pascal
If BarStatus(1) = 2 Then
Begin
    CurrentTradeVolume  = 0;
    PreviousTickCount   = 0;
    BigBlockVolume      = 0;
    BigBlockPercentage  = 0;
End
Else
    PreviousTickCount = Ticks;
```

`BarStatus(1) = 2` signals that the current bar has just closed. All accumulators are reset in preparation for the next bar. Crucially, `PreviousTickCount` is only updated when the bar is **not** closing — the `Else` branch ensures that on a bar-close tick, the reference is reset to `0` rather than being set to the final tick count of the closed bar, which would create a false delta on the first tick of the new bar. See the v1 vs v2 section for the original formulation of this reset and why the order matters.

### 7. Reference Plots

```pascal
Plot2(AlertThresholdPct, "Threshold");
Plot3(0, "ZeroLine");
```

`Plot2` and `Plot3` are placed outside the `If LastBarOnChart` block, so they draw across all historical bars. This is intentional — the threshold line and zero baseline provide a consistent visual reference in the indicator subgraph regardless of whether historical data is being displayed.

## v1 vs v2: The Difference

Both versions implement the same detection logic. The evolution addresses naming clarity and a subtle initialization order issue.

### v1 — Functional but Order-Dependent Reset

v1 updates `LastV` (the tick reference) and then resets all variables on bar close in the same sequential block:

```pascal
LastV = Ticks;

If Barstatus(1) = 2 Then
Begin
    TradeV    = 0;
    LastV     = 0;
    BigTradeV = 0;
End;
```

The net result is correct — `LastV` ends up at `0` when the bar closes — but only because the reset executes after the assignment. The code depends on execution order rather than expressing intent directly. If the two blocks were reordered without understanding this dependency, a bug would be introduced silently: `PreviousTickCount` would carry the final tick count of the closed bar into the first tick of the new bar, generating a large spurious delta on the new bar's first trade classification.

### v2 — Explicit `Else` Branch for Unambiguous Intent

v2 restructures the reset using an `Else` branch that makes the two cases mutually exclusive:

```pascal
If BarStatus(1) = 2 Then
Begin
    CurrentTradeVolume  = 0;
    PreviousTickCount   = 0;
    BigBlockVolume      = 0;
    BigBlockPercentage  = 0;
End
Else
    PreviousTickCount = Ticks;
```

`PreviousTickCount` is now only updated when the bar is alive. On bar close, it resets. The logic is self-documenting: a reader does not need to trace execution order to understand what happens in each case.

v2 also removes the redundant `TotalTicks` intermediate variable present in v1, which was always equal to `Ticks` at the point of use and added no information. The division uses `Ticks` directly.

**Summary:**

| | v1.0 | v2.0 |
|---|---|---|
| **Tick delta estimation** | ✓ | ✓ |
| **Block volume accumulation** | ✓ | ✓ |
| **Alert on threshold** | ✓ | ✓ |
| **`Else` branch for reset (unambiguous)** | — | ✓ |
| **`TotalTicks` intermediate variable** | ✓ (redundant) | ✗ removed |
| **Descriptive variable names** | `TradeV`, `LastV`, `BigTradeV`, `BigBlockPct` | `CurrentTradeVolume`, `PreviousTickCount`, `BigBlockVolume`, `BigBlockPercentage` |
| **Descriptive input names** | `BigblockTradeSize`, `ThresholdAlertPct` | `BigBlockTradeSize`, `AlertThresholdPct` |

## Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `BigBlockTradeSize` | 1000 | Minimum tick delta required to classify an update as a block trade. Updates with a delta below this value are treated as retail or noise activity. |
| `AlertThresholdPct` | 66 | Percentage of total tick activity that must come from block trades to trigger an alert. A value of 66 means block trades must account for two thirds or more of all tick activity on the bar. |

**On calibrating `BigBlockTradeSize`:** The appropriate threshold depends on the instrument and its typical tick activity patterns. On ES futures during active sessions, a delta of 1,000 may capture only the largest institutional prints; on thinner instruments or quieter sessions, a lower value may be more appropriate. This parameter requires empirical calibration against the target instrument's historical tick behavior.

**On calibrating `AlertThresholdPct`:** A threshold of 66% implies that block trades are not merely present but dominant — two out of every three tick updates have been classified as large. Lower values will produce more frequent alerts with potentially lower signal quality; higher values filter for only the most concentrated block activity.

## Interpretation Scenarios

### Scenario A — Block Activity Spike at Session Open

- ES opens. First tick arrives. `PreviousTickCount = 0` — guard condition blocks delta calculation.
- Tick 2: `Ticks = 3`, `PreviousTickCount = 1`. `CurrentTradeVolume = 2`. Below `BigBlockTradeSize`. Normal retail flow.
- Tick 8: `Ticks = 1,050`, `PreviousTickCount = 47`. `CurrentTradeVolume = 1,003 >= 1,000`. Block trade classified. `BigBlockVolume = 1,003`.
- `BigBlockPercentage = (1,003 / 1,050) * 100 = 95.5%`. Exceeds `AlertThresholdPct = 66`.
- **Alert fires.** A large price update cluster dominated the bar's tick activity at the open.

### Scenario B — Distributed Retail Activity, No Alert

- Midday session. Tick deltas consistently arrive in small clusters: 2, 5, 3, 8, 4. All below `BigBlockTradeSize = 1,000`.
- `BigBlockVolume` remains at `0`. `BigBlockPercentage = 0%`.
- No alert. The bar reflects fragmented, retail-scale tick activity.

### Scenario C — Mixed Activity, Threshold Not Reached

- Early afternoon. Several block-classified updates accumulate: `BigBlockVolume = 800`.
- Total ticks at the same point: `Ticks = 1,600`.
- `BigBlockPercentage = (800 / 1,600) * 100 = 50%`. Below threshold of `66%`.
- No alert. Block activity is present but not dominant enough to trigger.

### Scenario D — Bar Close and Clean Reset

- Bar closes. `BarStatus(1) = 2`. All accumulators reset to `0`. `PreviousTickCount = 0`.
- First tick of the new bar: `PreviousTickCount = 0` → guard condition blocks delta calculation. No spurious classification.
- Second tick: `PreviousTickCount > 0` → delta calculated normally. Clean start to the new bar.

## Key Features

- **`IntrabarPersist` accumulation:** All state variables retain their value across tick updates within the same bar, enabling within-bar tracking of tick activity composition without recalculating from scratch on every tick.
- **`LastBarOnChart` scope:** The detection logic runs only on the live bar. Historical bars are untouched, making the indicator computationally efficient and unambiguous about its real-time-only purpose.
- **Tick delta as trade proxy:** By differencing consecutive `Ticks` values, the indicator approximates the intensity of each price update without requiring access to individual trade records or L2 data.
- **`Else`-branch reset (v2):** The bar reset and tick reference update are mutually exclusive by construction, eliminating the order-dependent fragility present in v1.
- **Reference plots on historical bars:** `Plot2` (threshold) and `Plot3` (zero line) draw across all bars, providing a stable visual baseline in the subgraph independent of where the live calculation is occurring.
- **Alert on every qualifying tick:** The alert condition is evaluated on each tick update, meaning the indicator responds immediately when the threshold is crossed rather than waiting for bar close.

## Limitations

### `Ticks` as a Proxy — What the Indicator Actually Measures

The most important limitation of this indicator is the interpretation of `Ticks`. In EasyLanguage, `Ticks` counts **price update events within the bar** — each time the price changes, `Ticks` increments by 1. It does not measure the number of contracts or shares traded in each event.

The consequence is that `CurrentTradeVolume = Ticks - PreviousTickCount` is not a count of contracts per trade. It is a count of how many price updates occurred between two consecutive indicator evaluations. This distinction has several practical implications:

- A single institutional order for 5,000 contracts may arrive as one tick update (one print, one price change) or as a burst of many smaller prints across multiple price levels. In the first case, a single delta of `1` would not reach `BigBlockTradeSize = 1,000` even though the order was enormous. In the second case, the burst might generate many small deltas, none of which individually exceed the threshold.
- Two consecutive retail orders at the same price may register as a single tick update if no price change occurs between them, effectively appearing as one combined event.
- Session-open and news-driven periods typically compress many large trades into rapid successive tick updates, which can inflate `CurrentTradeVolume` even when individual orders are not unusually large.

The indicator is therefore best understood as measuring **tick update density and clustering** rather than literal block trade volume. Elevated readings signal that tick activity is arriving in concentrated bursts rather than uniformly — a pattern statistically associated with large-order activity, but not a direct measurement of it.

### Accessing Real Contract Volume — L2 Data Requirement

Accurately measuring the volume of contracts behind each individual trade requires access to **Level 2 (L2) market data**, also known as market depth or time-and-sales data at the individual trade level. L2 data provides a record of each executed trade with its exact size in contracts or shares, the price, and the timestamp — the raw input needed to classify individual prints as block trades by contract count rather than by tick delta.

In TradeStation, accessing per-trade contract volume at the tick level requires a data subscription that includes time-and-sales detail beyond standard OHLCV bars. Most standard data plans provide bar-level volume aggregates; individual trade records are typically available through premium tick data subscriptions or direct exchange feeds. Other platforms and brokers have analogous tiering: standard subscriptions aggregate volume at the bar level, while L2 or tick-data subscriptions expose the individual trade records that would make true block trade detection possible.

Until L2 data is available, `Ticks`-based delta estimation remains a computationally accessible approximation — useful for detecting concentrated activity, provided the user understands it is measuring price update frequency rather than contract flow.

## Use Cases

**Intraday session monitoring on equity index futures:** On ES or NQ, block trade activity tends to concentrate at session opens, around economic data releases, and during directional trend days driven by institutional positioning. The indicator provides a real-time signal when tick activity shifts from the fragmented pattern typical of retail flow toward the burst pattern associated with large orders.

**Pre-breakout activity detection:** Block trade clustering often precedes or accompanies price breakouts from intraday ranges. An elevated `BigBlockPercentage` while price is compressing near a key level can serve as early confirmation that institutional participants are active before the move is visible in price alone.

**Filter for other signals:** The indicator can act as a volume-quality filter for other entry signals — a crossover or pattern signal that coincides with high block activity has a different character than the same signal during thin, retail-dominated conditions.

**Timeframes:** The indicator is meaningful only on timeframes where intrabar tick dynamics are visible — 1-minute or faster bars during active sessions. On daily bars, within-bar tick behavior is not accessible, and the indicator's real-time-only design makes it unsuitable for historical analysis.

**Instruments:** Best suited to liquid futures markets (ES, NQ, CL, GC) where institutional participation is significant and tick update bursts are a meaningful signal. In illiquid instruments, sporadic tick updates can generate large deltas from thin trading rather than genuine block activity, producing false positives.

## Considerations for a Future v3

### 1. Formalised Bar Lifecycle Block

v2 addresses the order-dependent reset of v1 with an `Else` branch. A v3 refinement would make the intent even more explicit by enclosing the `Else` branch in its own `Begin`/`End` block and grouping all bar-lifecycle logic under a single labeled region:

```pascal
{--- Bar lifecycle management ---}
If BarStatus(1) = 2 Then
Begin
    {--- Bar closed: reset all accumulators ---}
    CurrentTradeVolume  = 0;
    PreviousTickCount   = 0;
    BigBlockVolume      = 0;
    BigBlockPercentage  = 0;
End
Else
Begin
    {--- Bar active: update tick reference ---}
    PreviousTickCount = Ticks;
End;
```

While EasyLanguage does not require `Begin`/`End` for a single-statement `Else`, the explicit delimiters signal to any future reader that the two branches are deliberate counterparts — not an assignment that happens to precede a reset.

### 2. Real Contract Volume via L2 Data

A v3 that addresses the core measurement limitation would replace the `Ticks` delta with actual per-trade contract volume sourced from L2 or time-and-sales data. In TradeStation, this would require either a premium tick data subscription exposing individual trade records via `TradeVolume` or an equivalent time-and-sales function, or an external data feed integrated via DLL or external function for exchange-level trade data access.

With real per-trade volume available, `BigBlockTradeSize` would become a true contract threshold — classify any single trade of 500 contracts or more as a block — and `BigBlockPercentage` would express the fraction of bar volume transacted in block-sized prints. This would make the indicator a genuine institutional footprint detector rather than a tick density monitor.

### 3. Per-Bar Alert Throttling

The current alert fires on every qualifying tick, which can generate repeated alerts within a single bar when block activity remains persistently elevated. A v3 improvement would introduce a per-bar alert flag to limit the alert to one fire per bar:

```pascal
IntrabarPersist AlertFiredThisBar(False);

If BigBlockPercentage > AlertThresholdPct and not AlertFiredThisBar Then
Begin
    Alert("High block trade activity detected");
    AlertFiredThisBar = True;
End;
```

`AlertFiredThisBar` would reset alongside the other accumulators on bar close, ensuring each bar can generate at most one alert — reducing noise without losing the signal on the bar where the threshold is first crossed.
